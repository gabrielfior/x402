x402 × ERC-8004 — ticket flow

1-step x402 (same URL). Proposed — not implemented.

Summary — what to build

  • New: TicketMinter — settle + mint ticket; finalize response; recover by txHash.
  • Upgrade: ReputationRegistry — ticket-gated giveFeedbackWithTicket, on-chain dedup, agent disputes; legacy giveFeedback disabled.
  • SDK: facilitator routes settle → minter; agent-server runs handler then triggers finalize; client submits feedback.
Client — payer, digests, feedback tx Agent server — 402, route handler, receipt sig Facilitator — verify, settle/mint/finalize gas Chain — TicketMinter, IdentityRegistry, ReputationRegistry
Ticket: NONE → MINTED (paid + request) → FINALIZED (response) → CONSUMED (feedback)

Smart contracts

TicketMinter new
settleAndMintTicket — facilitator submits; runs scheme settlement (EIP-3009 / Permit2 / ERC-20); on success writes ticket (MINTED) with payer, agentId, requestHash, interactionHash, endpoint, settlementTxHash. Revert ⇒ no ticket.
finalizeTicket — sets responseHash, optional agent receipt sig; FINALIZED.
ticketIdByTxHash — recovery if client never got ticketId in HTTP response.
ReputationRegistryUpgradeable upgrade
giveFeedbackWithTicket only — requires finalized, unconsumed ticket; msg.sender == payer; dedup (agentId, payer, txHash) and (agentId, payer, feedbackHash); consumes ticket.
disputeFeedback — agent owner/operator only.
giveFeedback — revert (no bypass).
IdentityRegistry
Unchanged — ownerOf(agentId) for payTo binding and receipt/dispute auth.

Key considerations

  • No IPFS required. Payment and job binding live on-chain in the ticket (requestHash at mint, responseHash at finalize). feedbackURI is optional (rich text only).
  • vs off-chain aggregator only. Aggregators can still verify, but anyone can spam giveFeedback today and composable contracts cannot trust entries without log/IPFS work. Tickets make “paid for this job” a cheap on-chain read; dedup is enforced in the registry, not filtered downstream.
  • Feedback connected to jobs. A ticket is the job record: one successful settlement ⇒ one mint per (payer, agentId, txHash). TicketMinter is the singleton choke point for “payment succeeded” (revert ⇒ no ticket). The buyer still calls giveFeedbackWithTicket as msg.sender (correct attribution); the registry gates on the ticket instead of a separate gateway submitting on the buyer’s behalf.

Phases (one paid HTTP request)

0 — Discover client agent server x402

Client GET /resource → agent server returns 402 + extensions.erc8004.agentId, accepts. Client learns endpoint (this URL) from the server.

1 — Bind request client

Client computes requestHash, interactionHash for the ticket (erc8004). Payment signing (EIP-3009 / Permit2 / ERC-20) is standard x402 — see phase 2.

2 — Sign payment client x402

Standard x402: client signs token authorization from PaymentRequired (EIP-3009 / Permit2 / ERC-20).

3 — Paid retry client agent server facilitator x402

Client retries same URL with X-PAYMENT. Agent server middleware calls facilitator verify → settle.

4 — Settle + mint facilitator chain on-chain

Facilitator tx: TicketMinter.settleAndMintTicket. Atomic transfer + MINTED ticket. Facilitator pays gas.

5 — Serve resource agent server x402

Agent server route handler runs only after mint succeeds (middleware gate). Executes business logic, builds HTTP response, computes responseHash from response digests. No chain tx in this step.

6 — Finalize agent server facilitator chain on-chain

Agent server (or client) supplies responseHash + optional receipt sig; facilitator submits finalizeTicket → FINALIZED. Agent may refuse to sign; finalize without sig still allowed.

7 — HTTP response agent server client x402

Agent server → client: 200, body, PAYMENT-RESPONSE (txHash, payer, ticketId). Optional X-X402-Interaction-Receipt.

8 — Feedback client chain on-chain

Client tx: ReputationRegistry.giveFeedbackWithTicket. Requires FINALIZED ticket; dedup; consumes ticket. Client pays gas. Agent may disputeFeedback later.
