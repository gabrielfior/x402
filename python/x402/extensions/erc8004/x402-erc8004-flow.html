⚙ Standard x402 (what the SDK gives us)

  • Payment flow — client gets 402, signs EIP-3009 (USDC) or Permit2 (any ERC-20), retries, facilitator verifies + settles
  • x402Client / x402Facilitator / x402ResourceServer — core classes for each party
  • ExactEvmServerScheme / ExactEvmFacilitatorScheme — EIP-3009 + Permit2 support
  • PAYMENT-RESPONSE — settlement txHash + payer returned to client
  • Extension hooks — enrich_payment_required_response, enrich_settlement_response

✨ Our additions (ERC-8004 reputation layer)

  • No contracts owned — client calls the standard ReputationRegistry.giveFeedback directly; FeedbackGateway removed
  • Canonical artifact — JSON binding paymentRequirements + paymentPayload + responseDigest + txHash, committed via feedbackHash / feedbackURI
  • Scheme-agnostic — verification keys off the universal ERC-20 Transfer event (EIP-3009, Permit2, plain ERC-20 all work)
  • Optional agent receipt — server signs an InteractionReceipt over {version, settlement, request, response} (digests) and returns it in the X-X402-Interaction-Receipt header; commits the agent to what it served; absence lowers trust tier, never blocks
  • Direct submission — plain type-2 tx; msg.sender on the registry is the payer (correct attribution, no EIP-7702)
  • Off-chain dedup — aggregators key on (payer, agentId, txHash), latest block wins
  • Censorship-resistant — no gatekeeping contract; the agent cannot block feedback submission

Flow steps — standard vs new

1
Client → Resource Server x402
GET /resource. No payment header. Server returns 402 + payment requirements and extensions.erc8004 = { agentId }.
2
Client → Facilitator x402
Client signs the token authorization (EIP-3009 for USDC, Permit2 for any ERC-20). POST /verify validates signature + simulates.
3
Client → Resource Server x402
Retries with the X-PAYMENT header. Middleware intercepts and triggers settlement.
4
Server → Facilitator → On-chain x402
POST /settle. Facilitator executes the transfer; the asset emits a standard ERC-20 Transfer(payer, payTo, amount) log.
5
Server → Client new
200 OK + PAYMENT-RESPONSE (settlement txHash, payer). Optionally, once the response is known, the server signs an InteractionReceipt at the HTTP layer (create_interaction_receipt) and returns it in the X-X402-Interaction-Receipt header — signature over keccak256("x402-erc8004-receipt" + chainId + txHash + interactionHash), where interactionHash = keccak256(canonical({version, settlement, request, response})). This commits the agent to the exact request it answered and response it returned.
6
Client builds + publishes artifact client
Client assembles the canonical artifact { settlement, interaction, feedback } (embedding the receipt at interaction.response.agentSignature if present), computes feedbackHash = keccak256(canonical(artifact)), and uploads the bytes to obtain feedbackURI (IPFS/Arweave in production).
7
Client → On-chain (pays own gas) new
Client sends a plain type-2 tx calling ReputationRegistry.giveFeedback(agentId, value, …, feedbackURI, feedbackHash). No gateway, no EIP-7702 delegation, no relayer — msg.sender on the registry is the payer EOA, so attribution is correct by construction.
8
Aggregator verifies (off-chain) verifier
Any reader of NewFeedback events fetches feedbackURI, checks keccak256(bytes) == feedbackHash, confirms the ERC-20 Transfer in txHash matches payer/payTo/amount, checks IdentityRegistry.ownerOf(agentId) == payTo, then assigns a trust tier and dedups on (payer, agentId, txHash).

Detailed: the canonical artifact & its two hashes

All request↔feedback binding lives off-chain in one JSON document, anchored on-chain by a single bytes32. The agent commits only to the payment-level interaction; the client's rating is its own.

A
interaction_hash — agent-signed new
keccak256(canonical({ version, settlement, request, response })), where response excludes its own agentSignature (no signing over itself). Covers the request the agent answered and the response it produced (digests) — so the agent cannot later repudiate its output. Excludes only the client's feedback (the rating is the client's opinion). Server and client compute the identical preimage, so the receipt verifies.
B
feedback_hash — on-chain commitment v1
keccak256(canonical(full artifact)) — covers settlement, request/response digests, the client's feedback, and the embedded agentSignature. This is the value passed to giveFeedback and hosted (by preimage) at feedbackURI.
C
Trust tiers verifier
FULL — all checks pass + valid agent receipt. CLIENT_ONLY — payment proven, no receipt (response is client-claimed). DISPUTED — agent receipt present but mismatched. REJECTED — integrity or chain checks failed.
